Progetto firefly3SYNC | Lab ISS 2026

Introduction

Realizzazione di un simulatore di un sistema costituito da 3 lucciole. Il comportamento evolve in due fasi:
  1. Primi 10 secondi: lampeggio a frequenza costante propria (asincrono).
  2. Successivamente: lampeggio in maniera sincrona.

Requirements

Definire un modello QAK di un sistema firefly che simuli accuratamente il ciclo di vita e la sincronizzazione di tre attori lucciola.

Requirement Analysis

  • Identità: Il sistema si chiama firefly3SYNC.
  • Comportamento Asincrono: Nei primi 10s, ogni firefly (attore) lampeggia con un timer interno indipendente.
  • Sincronizzazione: Dopo la fase iniziale, gli attori devono allineare i propri cicli di accensione.

Problem Analysis

Il cuore del problema è la gestione del tempo. Abbiamo modellato il ciclo come un'alternanza di stati:

State flash {
    [# Timer = java.util.Random().nextLong(500L, 1000L) #]
    // Turn On
    println("$name | turnOn $Timer")
    delayVar Timer
    // Turn Off
    println("$name | turnOff $Timer")
}

La ripetizione è garantita da una transizione ricorsiva temporizzata:

Transition t0 whenTime 500 -> flash

Project & Design

Miglioramento della visibilità: Per monitorare lo stato abbiamo introdotto l'attore griddisplay.

Le lucciole ora sono reattive: non decidono autonomamente ma attendono un segnale globale di sincronizzazione.

Attore Firefly (Reattivo)

State wait_sync { }
Transition t0 whenEvent sync -> flash

State flash {
    forward griddisplay -m cellstate : cellstate($X, $Y, 1)
    delay 500
    forward griddisplay -m cellstate : cellstate($X, $Y, 0)
}
Goto wait_sync

Attore Coordinator (Master Clock)

Il coordinator emette un Event (broadcast), garantendo che tutte le lucciole scattino nello stesso istante logico.

State loop {
    emit sync : sync(1)
    delay 1500
}
Goto loop

Maintenance

Il sistema è estensibile: per aggiungere lucciole è sufficiente aumentare il numero di attori nel modello QAK e mapparli sulla griglia del griddisplay.
